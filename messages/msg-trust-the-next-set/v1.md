# Message `msg-trust-the-next-set` v1: Earn trust in the next set

## Target situation

A U.S. male lifter in his 20s or 30s trains alone for muscle or strength. He can perform the exercises and may already have a Workout, but he still has to decide what Weight and Reps should come next or whether the current Program is progressing.

The most relevant moment is after he has accumulated performance history and before a Working Set or the next Workout, when he must translate past Weight, Reps, RIR, equipment increments, and recent success or failure into one next action.

## Problem pattern

The lifter may check prior Sets, remember how hard they felt, compare Rep Range, account for the smallest available plate or machine increment, and then choose whether to add Weight, add Reps, maintain, or reduce. That judgment repeats across exercises, sessions, and weeks.

When the next action is based on memory, impulse, or an unexplained automated number, the lifter faces two different trust problems:

- trusting his own inconsistent decision enough to commit to the Set;
- trusting an app recommendation whose inputs and constraints he cannot see.

The planned product must not replace visible uncertainty with opaque authority. It must turn recorded performance into a recommendation whose behavior remains understandable and editable.

## Belief shift

### From

Real progression means pushing harder whenever possible, and an automated recommendation is only useful if I accept a number I cannot independently judge.

### To

Progression is controlled force: increase when performance and loading constraints support it, maintain or reduce when they do not, and keep the actual Set as the final truth. A recommendation earns trust when it stays connected to visible inputs, respects real equipment, explains unexpected changes, and adapts to what I actually perform.

## Persuasion logic

1. Strength requires aggression in execution, but progression requires restraint in decision-making.
2. The next Weight and Reps depend on more than motivation: previous performance, current RIR, target Rep Range, exercise behavior, available increments, and Program policy can constrain the next useful action.
3. Applying those considerations repeatedly is bookkeeping the lifter may not want to manage across every exercise and week.
4. Blind automation does not remove the trust problem; it only hides the judgment.
5. A credible system should prefill one next action while keeping the decisive inputs, unexpected-change reason, and actual recorded performance available.
6. LIFT CODE is being built to provide that controlled decision surface so the user can bring aggression to the Set instead of improvising the progression rule.

The claim that this mechanism will cause users to trust or follow LIFT CODE recommendations is a message hypothesis. It has not been validated with LIFT CODE users.

## Resistance and response

| Resistance | Response |
| --- | --- |
| I already know when to add weight. | The message does not deny the lifter's judgment. It makes the repeated cross-exercise bookkeeping and equipment constraints visible, then positions the recommendation as an editable default rather than a command. |
| An algorithm cannot know how I feel today. | LIFT CODE is not claimed to observe unentered readiness, pain, or recovery. The planned recommendation uses the performance and configuration inputs the product actually records, and the user records the real Set when it differs. |
| Maintaining or decreasing weight means the system is holding me back. | Controlled progression includes Increase, Maintain, and Decrease. The message should connect restraint to the available performance evidence without claiming every conservative decision is correct. |
| Recommendation explanations are just excuses for a bad algorithm. | Explanation alone does not prove quality. It makes inputs and constraints inspectable so recommendation behavior can be judged rather than accepted as mysterious authority. |
| This sounds like another complicated coaching app. | The complexity belongs inside the decision system. The planned workout surface prefills Weight and Reps and keeps deeper reasoning available on demand rather than making the user manage it before every Set. |
| The product does not exist yet. | Say so. Present LIFT CODE as being built and discuss the design standard it must meet; never imply availability, users, or proven outcomes. |

## Product role

LIFT CODE is being built to generate or accept a Program, read actual training performance, respect selected Gym equipment and increments, and prefill the next Weight and Reps. It is planned to show the key reason for unexpected Increase, Maintain, or Decrease outcomes, expose deeper inputs and methodology on demand, and let the user record a different actual result.

In this message, LIFT CODE does not command blind obedience. It is the planned system that must earn repeat trust by aligning recommendation, explanation, constraint, and actual performance.

## Evidence and limits

### Supported

- Product direction and claim boundaries: `context/product.md`.
- Public competitor-community language shows demand for history-based Weight recommendations applied to existing routines and describes weekly progression management as “tedious bookkeeping”: `context/user-language.md`.
- The product specification plans to use previous performance, current Weight, Reps and RIR, target Rep Range and RIR, exercise characteristics, equipment increments, and Program progression policy.
- The product specification plans editable recommendations and inspectable reasons and inputs.

### Unverified

- The primary audience will recognize recommendation trust as its strongest reason to follow the account or later use the product.
- Visible inputs and concise reasons will materially increase recommendation acceptance.
- The planned algorithm will produce recommendations users judge as accurate, safe, or superior.
- “Controlled aggression” or “restrained wildness” will outperform a more neutral training identity.
- The target user experiences progression bookkeeping frequently enough to switch products or pay.

### Message-specific limits

- Use trust as something the system must earn, not something the brand declares it already possesses.
- Use `being built`, `designed to`, or equivalent pre-development language for product capabilities.
- Do not call the recommendation scientific, validated, accurate, personalized, optimal, or proven without future direct evidence supporting that exact claim.
- Do not imply the product measures pain, fatigue, recovery, readiness, form, or injury unless a verified implemented input supports the statement.
- Do not promise plateaus will end, every Set will improve, or users will become stronger, larger, more attractive, or more confident because of the app.
- Masculinity and physical ambition may create relevance, but do not degrade, humiliate, or exclude people outside the primary audience.
